# Pool bots ÔåÆ N:1 (each pool bot serves multiple companies)

## Context

A founder hit `"No pool bot available and no global fallback configuredÔÇª"` on production. Investigation traced it to `resolvePoolAssignment()` returning `'pool_exhausted'` ÔÇö the prod pool has 5 bots, all already 1:1-assigned to 5 companies, and PR #41 (2026-05-06) deliberately removed the `TELEGRAM_BOT_TOKEN` global overflow.

The 1:1 enforcement (`UNIQUE INDEX ÔÇª ON baget_bot_pool(assigned_agent_group_id)`) was conservative: it covered the worst-case "one founder runs N companies through the same bot" scenario where Telegram's `(bot, user) ÔåÆ one-DM` rule collapses chats. For the typical case ÔÇö N different founders or group chats keyed by `chat_id` ÔÇö that constraint adds nothing.

## Change

Switch the bot pool from 1:1 to N:1 via a junction table. Each pool bot can now serve multiple companies; routing already happens downstream via `messaging_group_agents` (`chat_id ÔåÆ agent_group`). The same-founder edge case is mitigated best-effort by a new preference in the allocator.

## What changed

1. **Migration 020** (`src/db/migrations/020-baget-bot-pool-assignments.ts`): creates `baget_bot_pool_assignments(agent_group_id PK, bot_username FK, assigned_at)`, backfills from the legacy 1:1 column, drops the UNIQUE index + orphan trigger + obsolete columns (`assigned_agent_group_id`, `assigned_at`, `status`), replaces `status` with nullable `retired_at` (NULL = active in rotation).
2. **`src/db/baget-bot-pool.ts` rewrite**: 
   - `assignNextAvailableBot(agentGroupId, ownerUserId?)` ÔÇö least-loaded pick with same-founder preference via SQL.
   - `releaseBot(agentGroupId)` ÔÇö DELETE the junction row; bot stays in the pool.
   - `getBotPoolEntryByAgentGroup` ÔÇö JOIN through the junction.
   - New `countAssignmentsForBot(username)` helper for observability.
   - `BotPoolRow` type drops 3 fields and gains `retired_at`.
3. **`src/baget-admin-server.ts`**: threads `ownerUserId` through `resolvePoolAssignment` ÔåÆ `assignNextAvailableBot`. Error messages updated: `'pool_exhausted'` is now an operator-misconfig signal (zero active bots), not a capacity issue. Admin `/baget/bot-pool/export` replaces `_meta.assignedAgentGroupId` with `_meta.assignmentCount`.
4. **Tests**: `db/baget-bot-pool.test.ts` and `baget-bot-pool-integration.test.ts` rewritten/expanded for N:1. New describe blocks: "N:1 allocation policy", "N:1 multi-tenant scenarios". Migration 016 doc-comment marked superseded.

## What did NOT change

- Per-bot webhook secrets, `webhook_registered_at` register-once, channel adapter outbound paths (they just read `bot_token_value`), `messaging_group_agents` chat-id routing, the legacy Vela `cfg.telegramBotToken` fallback path, `seedBotPoolEntry` rotation semantics, `setMyName` (already removed 2026-05-05 ÔÇö operator owns names via BotFather).

New tests cover: 2 different agent_groups sharing one bot, idempotent re-pair on junction PK, least-loaded distribution, FIFO tiebreak, same-founder preference (2 companies same founder ÔåÆ different bots), retired-bot exclusion, FK CASCADE replacing the orphan trigger, 3 founders pairing successfully against a 2-bot pool.
